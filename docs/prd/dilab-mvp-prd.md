## 0. 한 문장 목표

**도메인 전문가 리뷰 DB + 공개 리뷰** 를 벡터 DB 에 넣고 RAG 로 LLM 에 결합해, *전문가 출처를 명시하는* 제품 평가 리포트를 생성하는 도메인-플렉시블 프로토타입.

---

## 1. DILAB 큰 기능 단위 (6개) + 관통 특성 (2개)

> 싱클리·인핸스·포켓서베이의 기능을 그대로 따라 만들지 않는다. **큰 기능 단위로 6개**, 그리고 전체를 관통하는 차별화 특성 2개로 정리한다.

### 1.1 P0 — MVP 핵심 기능 6개

| # | 큰 기능 | 구성 모듈 | 동작 OK 기준 | 벤치마크 매핑 |
|---|---|---|---|---|
| **B1** | **데이터 파이프라인** | (a) 공개 리뷰 수집 — **AI Hub 한국어 리뷰 데이터셋 + 앱스토어/구글플레이 리뷰** (합법·안정 경로) (b) 전문가 DB 수집 (업로드·URL) (c) 청킹·임베딩·벡터 DB 저장 | 두 소스의 데이터가 동일 인덱스에 들어가고 메타데이터(작성자·도메인·날짜·신뢰도)가 붙어 있으면 OK. 쿠팡·네이버쇼핑 등 *상업적 크롤링은 약관·법무 검토 후* PMF 검증 단계에서 결정 | 싱클리 F2·F10 |
| **B2** | **토픽 분석 (BERTopic)** | BERTopic 으로 리뷰 묶음에서 토픽 자동 발굴 → 토픽별 키워드·문서 수·대표 인용 추출 | 도메인 코퍼스 ≥500건에 대해 토픽 ≥10개가 의미 있게 나뉘면 OK (사람이 읽고 라벨 부여 가능 수준) | 싱클리 F4 부활 |
| **B3** | **분류·감성 분석** | (a) 도메인 YAML 기반 카테고리 자동 분류 (예: 화장품 → 효능/성분/가격/사용감/안전성) (b) 한국어 감성 3분류(긍/부/중) + 강도 점수 | 리뷰 100건에 카테고리·감정 라벨이 붙고 카테고리×감정 분포가 그려지면 OK. 도메인 바꾸면 카테고리도 코드 변경 없이 바뀌면 OK | 싱클리 F3·F5 |
| **B4** | **사용자 여정 지도** | 리뷰·토픽을 **인지 → 구매 → 사용 → 재구매** 단계에 자동 매핑(LLM zero-shot 분류). 각 단계별 주요 토픽·감정·인용 노출. **재구매는 *의도* 추출이지 실제 구매 데이터 X** | 한 제품에 대해 여정 4단계가 모두 채워지고 각 단계의 핵심 이슈 ≥1개씩 추출되면 OK. **분류 정확도 목표 ≥80%**, 노이즈 라벨은 "추정" 표시 | 포켓서베이 CX 차용 (DILAB 신규 도입) |
| **B5** | **RAG Q&A + LLM 추천 (DILAB Ask)** | (a) 자연어 질문 입력 (b) 전문가 DB 우선 → 공개 리뷰 보조 벡터 검색 (c) LLM 합성 답변 + **출처 카드** ("전문가 N건 / 일반 M건") (d) 끝에 **LLM 추천**("이 제품은 X 측면에서 강점, Y 에서 약점 → A/B/C 중 X 추천" 형식). **5축 평가 점수의 *근거 hover 인용* 은 MVP 에선 가장 강한 1축·약한 1축 2개만** (LLM 호출 비용 통제) | 질문 1개에 출처 카드 + 추천 한 줄이 함께 나오면 OK. 인용 정합률 ≥95% (스폿 체크 10건) | 싱클리 F1 + DILAB 차별화 (LLM 추천은 신규) |
| **B6** | **리포트 + 대시보드 (Streamlit)** | (a) 한 제품 평가 리포트 1종 (요약 / 토픽 / 여정 / 감정 / 출처 / 추천) (b) Streamlit 인터랙티브 대시보드(도메인 선택·필터·차트) (c) Markdown·PDF export | 도메인 선택 → 리포트 + 대시보드가 5초 안에 렌더링되고 PDF 내보내기 동작하면 OK | 싱클리 F6·F12 |

### 1.2 P0 — 관통 차별화 특성 2개 (★)

| # | 특성 | 구현 위치 | 동작 OK 기준 |
|---|---|---|---|
| **★1** | **출처 카드 가시화** | B5 (RAG 답변)·B6 (리포트) 양쪽에 항상 노출 | 모든 답변·리포트에 "전문가 N건 / 일반 M건" 카드가 붙고 클릭 시 원문 인용으로 이동하면 OK |

## 2. 시스템 아키텍처 (개략)

```
┌────────────────────────────────────────────────────────────────────────────┐
│                                                                            │
│  [B1 수집·인덱싱]      [B2·B3·B4 분석 레이어]            [B5·B6 출력]      │
│                                                                            │
│  공개 리뷰 ─┐                                                              │
│            ├─→ 청킹·임베딩 ─→ 벡터 DB ─┬─→ BERTopic ───→┐                  │
│  전문가 DB ─┘   (도메인 메타)          ├─→ 분류·감성 ───┼─→ RAG + LLM      │
│                                        └─→ 여정 매핑 ───┘     │             │
│                                                               ↓             │
│                                                       답변 + 출처 카드      │
│                                                       + LLM 추천            │
│                                                               │             │
│                                              ┌────────────────┘             │
│                                              ↓                              │
│  설정 : domain.yaml                  Streamlit 대시보드                     │
│  (카테고리·키워드·여정 단계·소스 URL)  + 리포트 PDF/Markdown               │
│                                                                            │
└────────────────────────────────────────────────────────────────────────────┘
```

**구성요소 (M1 ADR 에서 확정)**

| 레이어 | 후보 | 선정 기준 |
|---|---|---|
| 수집 | Playwright MCP / Naver Search MCP | 이미 설치 완료. 추가 도구 없이 즉시 사용. |
| 임베딩 | OpenAI text-embedding-3-small / BGE-M3 / E5-large | 한국어 품질·비용 비교 후 선택 |
| 벡터 DB | Qdrant(로컬·무료) / Pinecone(매니지드) / MongoDB Atlas | 학생 인프라 적합성·비용 |
| LLM | Claude Haiku / GPT-4o-mini | 비용·한국어 품질 |
| 분류·감성 | KoBERT / klue-roberta-small / 또는 LLM 프롬프트 | 빠른 프로토타이핑 vs 정확도 |
| 토픽 모델링 | **BERTopic** (sentence-BERT + UMAP + HDBSCAN) | 한국어 임베딩 모델과 조합 검토 (KoSimCSE / BGE-M3 등) |
| 여정 단계 매핑 | LLM zero-shot 분류 또는 키워드 룰 | YAML 에서 단계 정의(인지/구매/사용/재구매)·각 단계별 트리거 키워드 |
| UI | Streamlit | 대학원생이 일주일 안에 만들 수 있는 수준 |

## 3. 미해결 사항 (3-5개로 축소)

| ID | 항목 | 어떻게 풀지 |
|---|---|---|
| OQ1 | **전문가 리뷰 DB 1차 코퍼스 확보** | 임상순 교수님 네트워크 → 화장품 도메인 전문가 1~2명 확보 가능 여부 확인. M2 시작 전. |
| OQ2 | **벡터 DB 선택** (Qdrant vs Pinecone vs Mongo) | M1 ADR — 비용·로컬 vs 매니지드·메타데이터 필터링 기능 비교 |
| OQ3 | **LLM·임베딩 모델 선택** | M1 ADR — 한국어 평가 데이터셋(KLUE 등)으로 빠른 벤치 1회 |
| OQ4 | **인핸스 Review Agent 후속 조사** | deep-research 1회 — 가격·고객사·자금. PRD 본격 작업 전 1시간이면 충분 |
| OQ5 | **BERTopic 토픽 결과의 평가 기준** | 사람이 토픽 라벨 부여 가능한지 정성 평가 + topic coherence 점수 등 정량 보조. M3 시작 시 결정 |
| OQ6 | **여정 단계 정의 (도메인별)** | 화장품 기준 "인지/구매/사용/재구매" 4단계 기본. 도메인 변경 시 YAML 에서 단계 자체를 바꿀 수 있게 설계 |
| OQ7 | **공개 리뷰 데이터 소스 합법성** | MVP 는 AI Hub + 앱스토어로 우회. 쿠팡·네이버쇼핑 같은 *상업 사이트 크롤링* 은 약관·법무 검토 후 PMF 단계에서 결정 |
| OQ8 | **디자인 모방 정책 (CLAUDE.md 2.1)** | 현재 Hard Rule 로 *싱클리 시각 모방 금지*. 정책 완화 시 교수님 협의 + CLAUDE.md 업데이트 필요. *현재 v2 인디고+머스타드 톤 유지* |

이외 GTM·페르소나·가격·메트릭 등은 *MVP 동작 검증이 끝난 뒤* 결정해도 늦지 않음.

## 3.1 Reality Check — 구현 가능성 (2026-05-26 추가)

화면이 약속한 것 중 *백엔드가 따라가기 가장 어려운* 3가지:

| 약속 | 현실 | MVP 조정 |
|---|---|---|
| 5축 레이더의 *축별 근거 hover 인용* | LLM 호출 5배·구조화 프롬프트 필요 | **가장 강한 1축 + 약한 1축, 총 2개만 인용 노출** |
| 여정 4단계 자동 분류 | LLM zero-shot 정확도 70~80% 한계 | 분류 신뢰도 ≥80% 목표 + 노이즈는 *"추정"* 표기 |
| "재구매율 52%" 같은 표기 | 실제 구매 데이터 X — *의도* 추출만 가능 | UI 표기 *"다시 사고 싶다는 사람"* 으로 정직하게 |

데이터 수집 경로 변경: **쿠팡·네이버쇼핑 크롤링 → AI Hub 한국어 리뷰 데이터셋 + 앱스토어 리뷰** (B1 반영).

## 6. 직접 경쟁 후보 (한 줄씩)

- **인핸스 Review Agent** — 기능 셋이 가장 유사. **DILAB 의 우위는 "전문가 DB + 출처 카드"** 두 가지뿐 — 이게 진짜 우위인지 후속 조사 필요 (OQ4).
- **싱클리** — 75만원/월 SaaS, B2B sales-led. DILAB 과 가격대·진입 형태가 달라 직접 충돌 적음.
- **포켓서베이 CX** — 데이터 소스가 설문(NPS) 이라 보완재 가능성.
