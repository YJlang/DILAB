# 3부. 10분 기술 설명 (학교 발표·R&D 제안용)

## 큰 그림 — 3 컴포넌트

```
[브라우저]  Next.js 16
              ↓ supabase-js (anon key)        Supabase Postgres + pgvector
              ↓                                  - documents, chunks (1024d vector)
              ↓ /api/ask, /api/analyze           - ratings.evidence_chunk_ids[]
              ↓                                  - citations 테이블
[Python 워커] FastAPI ─→ DeepSeek API           - HNSW index on embedding
              ↓ BGE-M3 (로컬)                    - RPC: match_chunks
              ↓ BERTopic (로컬)
              ↓ 네이버 검색 API
```

## 핵심 5 개념

### 1. RAG (Retrieval-Augmented Generation)
**비유**: 시험 볼 때 *오픈북* 으로 보는 것.

```
일반 LLM:      "이 제품 어때요?" → 모델이 기억으로만 답 → 환각·구식 정보 위험
RAG:           "이 제품 어때요?" → 1) 관련 청크 검색  2) 검색 결과 + 질문을 LLM 에 함께 → 답변
```

DILAB 의 RAG: BGE-M3 임베딩 → match_chunks RPC → top-K 가져옴 → DeepSeek 가 본문 안에 `[1][2]` 인용하며 답변.

### 2. 임베딩 (BGE-M3)
**비유**: 문장을 *1024 차원 공간의 점* 으로 변환. 의미 비슷한 문장끼리 가까워짐.

```
"민감성 피부에 좋은 토너"        → [0.12, -0.43, 0.07, ...] (1024개 숫자)
"자극 없는 클렌징 폼"            → [0.10, -0.45, 0.09, ...] (비슷한 점)
"가성비 좋은 카메라"             → [0.85,  0.21, -0.6, ...] (먼 점)
```

벡터 검색 = *공간에서 가장 가까운 점 찾기* (cosine similarity).

BGE-M3 는 한·영 동시 지원 + 1024 dim. Supabase pgvector 의 HNSW 인덱스에 저장 → 100만 벡터까지 빠른 검색.

### 3. BERTopic (토픽 모델링)
**비유**: 도서관 책 1000권을 *주제별 책장* 으로 자동 분류.

```
청크 1: "보습이 좋고 자극 없어요"     ─┐
청크 2: "어성초 함량 77% 가 핵심"    ─┤── Topic 0: "어성초·77·진정·토너"
청크 3: "트러블 피부에 추천"         ─┘

청크 4: "비건 클렌징 폼 분류 분석"   ─┐
청크 5: "세정력 vs 자극도 트레이드"  ─┤── Topic 1: "세정력·자극도·메이크업"
```

BGE-M3 임베딩 → UMAP (1024d → 3d 차원 축소) → HDBSCAN (밀도 기반 클러스터) → 토픽별 키워드 추출.

### 4. ★1 출처 카드 가시화 (DILAB 차별점 #1)
**비유**: 논문은 모든 주장에 *각주* 가 달려있음. DILAB 도 그렇게.

```
일반 AI 답변:   "이 제품은 민감성 피부에 적합합니다."         ← 근거 어디?
DILAB 답변:    "이 제품은 민감성 피부에 적합합니다 [1][3].   ← [1] = chunk_id ABC
               단 향에 민감하면 비추천이에요 [6]."             [3] = chunk_id DEF
                                                                [6] = chunk_id GHI
```

데이터 모델 단에서 강제:
- `ratings.evidence_chunk_ids uuid[]` — 5축 점수의 근거 청크
- `citations` 테이블 — 답변 ↔ 인용 청크 다대다
- *RAG 답변 + 5축 평가* 둘 다 출처 추적

### 5. ★2 도메인-플렉시블 (DILAB 차별점 #2)
**비유**: 같은 *분석 기계* 에 *재료* 만 바꿔서 다른 결과.

`domains` 테이블의 한 행:
```jsonb
{
  "slug": "cosmetics",
  "categories":     ["효능", "성분", "가격", "사용감", "안전성"],
  "rating_axes":    ["효능", "성분", "가격", "사용감", "안전성"],
  "journey_stages": [{"key":"aware","label":"처음 알게 됐을 때"}, ...]
}
```

새 도메인 F&B 추가 = SQL 1줄 + 코퍼스 업로드:
```jsonb
{
  "slug": "fnb",
  "categories":     ["맛", "신선도", "가격", "포장", "안전성"],
  "rating_axes":    ["맛", "신선도", "가격", "포장", "안전성"],
  "journey_stages": [{"key":"browse","label":"검색"}, ...]
}
```

→ 모든 화면 (S1~S6) 이 *자동으로* F&B 도 처리. UI 변경 0줄.

## 데이터 흐름 (1 cycle = 1분)

```
사용자 입력: "닥터지 레드 블레미쉬 크림"
   ↓ Next.js /api/analyze → ai-worker /analyze
   │
   ├─[1] 네이버 검색 API ──────────→ reviews 30 + expert 15 + shop 메타  (10s)
   ├─[2] build_slug ──────────────→ "drg-red-blemish-cica-soothing..."   (0s)
   ├─[3] products + documents insert ─→ Supabase (45 rows)               (1s)
   ├─[4] BGE-M3 embed_texts ───────→ 1024d × 45 vectors → chunks insert  (3s)
   ├─[5] label_domain ─────────────→ DeepSeek 45 호출 (분류·감성·여정)    (45s)
   └─[6] upsert_ratings ───────────→ 5축 + evidence_chunk_ids             (1s)
   ↓
페이지 리다이렉트 /products/<new-slug>
```

# 4부. 자주 묻는 질문 (FAQ)

**Q1. DeepSeek 정말 한국어 잘 해?**
A. 답변 품질 측정 결과 Claude Haiku 와 *체감 비슷*, 비용은 **약 10배 저렴** (입력 $0.27/M vs Claude $0.80/M). 정밀도 부족하면 코드 1줄 변경 (`LLM_MODEL`) 으로 Claude/GPT 로 swap. base_url 만 다른 OpenAI 호환 API.

**Q2. BGE-M3 가 임베딩 1024 차원인데 왜 1536 (OpenAI) 안 써?**
A. BGE-M3 는 **자체 호스팅 가능 + 무료 + 다국어 강함**. OpenAI text-embedding-3-small 은 API 호출당 비용 누적. 학생 MVP 예산 측면. 다만 정밀도 우선이면 변경 가능 (의존성 1개 + 차원 변경 마이그레이션).

**Q3. 데이터 수집은 합법인가?**
A. 네이버 *공식 API* 만 사용. 일 25,000회 quota. 학술·MVP 용도 안전. **상업화 단계** 에서 약관 재검토 필요. 쿠팡·올리브영 같은 *상업 사이트 크롤링* 은 회색지대 → PRD 의 OQ7 에 명시됨.

**Q4. RLS 실제 적용됐나?**
A. 16 정책 + advisor 권고 다 반영 (`auth.uid()` → `(select auth.uid())`). 단 현재 *익명 호출* (user_id NULL 허용). 본격 운영 시 Supabase Auth 활성화 → ask_queries.user_id 실사용.

**Q5. 환각 (hallucination) 은 어떻게 막아?**
A. 4단계 방어:
1. system prompt 에 "출처에 없는 내용 X" 강제
2. JSON 모드 — `{answer, recommendation}` 구조 강제
3. *답변 본문에 `[n]` 인용 번호* 표시 → 사용자가 수동 검증 가능
4. 5축 점수의 evidence_chunk_ids — *어느 청크로 점수 매겼는지* 추적

다만 *완전히* 방지 X. 약한 hallucination 은 발생 가능. UI 에 *"AI 합성, 근거는 출처 확인"* 디스클레이머 항상 노출.

**Q6. 임상순 교수님 연구실에서 만든 거 맞나?**
A. 네. AI 대학원생 윤준하 + 공동 검토 오현 학생. 협력사 대표의 정부 R&D 과제 수주 지원 목적.

**Q7. 다른 도메인 (F&B, 가전) 도 진짜 자동으로 되나?**
A. 데이터 모델 단에서 가능. domains 테이블의 categories/rating_axes/journey_stages 만 새 도메인용으로 정의 → 모든 화면 자동 적용. 단 *데이터 인제스션 소스* (네이버 검색 쿼리·전문가 큐레이션 등) 는 도메인별로 조정 필요.

# 5부. 발표 시 자주 받는 어려운 질문 + 답변 스크립트

### Q. "이거 너희가 직접 만든 거 맞아? 가져다 쓴 거 아니야?"
A. *"AI 모델은 가져다 썼고 (BGE-M3, DeepSeek, BERTopic — 모두 오픈소스 또는 API), **파이프라인과 데이터 모델은 직접 설계** 했어요. 특히 ★1 출처 추적과 ★2 도메인-플렉시블 두 가지는 데이터 스키마 단에서 강제하도록 설계했고, 이건 싱클리도 안 한 차별점이에요."*

### Q. "왜 한국어 특화 모델 안 쓰고 BGE-M3?"
A. *"한국어 전용 모델 (Solar, HyperCLOVA) 은 API 안정성·비용·접근성에서 부담이 있어요. BGE-M3 는 다국어 + 한국어 강함 + 자체 호스팅 무료 → 학생 MVP 에 최적. PMF 검증 후 한국어 모델 비교 평가는 PRD 의 OQ6 에 명시되어 있어요."*

### Q. "정부 R&D 제안서 키워드는 뭐야?"
A. *"PRD 부록 A 에 6개 정리되어 있어요: 도메인 특화 RAG / 휴먼 인 더 루프 LLM / 한국어 도메인 임베딩 / 다소스 신뢰성 가중 합성 / 도메인-플렉시블 데이터 인덱싱 / VoC 분석 자동화. 각 키워드의 학술 출처는 M1 ADR 단계에서 정리 예정."*

### Q. "싱클리랑 다른 점 다시 한 번?"
A. *"한 줄로는 — **양으로 답하는 싱클리, 질로 답하는 DILAB**. 싱클리는 15+ 채널을 자동 통합하는 *양 우선*. DILAB 은 전문가 큐레이팅 + 모든 답변에 출처 카드 명시하는 *질 우선*. 그리고 도메인-플렉시블 아키텍처는 싱클리도 산업별 별도 페이지로 분리하지만, DILAB 은 domain.yaml 한 줄로 새 도메인이 자동으로 모든 화면에 들어와요."*

# 6부. 한 페이지 요약 카드 (인쇄용)

```
─────────────────────────────────────────────────────────
                          DILAB
        질로 답하는 도메인 특화 RAG 제품 평가 서비스
─────────────────────────────────────────────────────────

  사용자 입력  →  네이버 자동 수집  →  BGE-M3 임베딩  →
  BERTopic 토픽  →  DeepSeek 분류·감성·여정  →
  5축 평가 (출처 카드 포함)  →  S1~S6 화면

  핵심 차별점 2가지:
  ★1 모든 답변·점수에 출처 chunk 추적 (citations 테이블)
  ★2 domain.yaml 한 줄로 새 도메인 자동 적용

  기술 스택:
  Backend  : Supabase Postgres + pgvector + RLS
```
